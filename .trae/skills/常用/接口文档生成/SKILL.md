# API 接口文档生成器

## 功能说明

根据代码、注释或现有接口定义，自动生成标准化的API文档（Swagger/OpenAPI格式），包含完整的参数约束说明和正确的请求/响应示例。

## 应用场景

- 从后端代码生成API文档
- 整理现有接口文档
- 生成可交互的API文档
- 导出标准格式的接口定义
- 生成完整的接口测试用例（Mock测试+单接口测试+流程测试）

## 支持格式

- Swagger 2.0
- OpenAPI 3.0
- Markdown格式文档
- HTML可交互文档（Swagger UI）

---

## 测试类型体系

API接口测试应包含三个层次，完整覆盖从单元到端到端的测试：

| 测试类型 | 目的 | 适用场景 | 测试粒度 |
|----------|------|----------|----------|
| Mock测试 | 模拟接口返回，隔离依赖 | 前端开发、接口未就绪 | 粗粒度 |
| 单接口测试 | 验证单个接口正确性 | 接口单元测试、参数校验 | 细粒度 |
| 业务流程测试 | 验证端到端业务流程 | 集成测试、业务验收 | 粗粒度 |

## 一、Mock测试

### 功能说明
模拟后端接口返回固定数据，无需真实后端服务即可进行前端开发和测试。

### 适用场景
- 后端接口未就绪
- 接口联调阶段
- 前端独立开发
- 异常场景模拟

### 生成内容
- Mock数据文件（JSON）
- Mock服务配置
- 请求/响应示例

### 文件位置
`测试全流程框架/10_api_test_接口测试/04_mock_server/`

### Mock数据规范
```json
{
  "/api/users/login": {
    "POST": {
      "200": {
        "message": "登录成功",
        "user": {
          "id": 1,
          "username": "admin",
          "name": "管理员"
        }
      },
      "400": {
        "message": "用户名和密码不能为空"
      },
      "401": {
        "message": "用户名或密码错误"
      }
    }
  }
}
```

## 二、单接口测试

### 功能说明
对单个API端点进行独立测试，验证接口的正确性，包括参数校验、边界值、错误处理、安全性等。

### 适用场景
- 接口单元测试
- 接口功能验证
- 参数校验测试
- 安全测试

### 测试方法综合应用

根据 `/Users/admin/Desktop/test-management-platform 2/.trae/skills/常用/测试用例/SKILL.md` 规范，单接口测试必须综合应用以下测试方法：

#### 1. 边界值分析法
- **适用场景**：数值输入（金额、数量）、字符串长度、日期范围
- **设计要点**：覆盖六点边界值（最小值-1、最小值、最小值+1、最大值-1、最大值、最大值+1）
- **CRM接口边界值示例**：

| 参数 | 边界条件 | 测试用例 |
|------|----------|----------|
| 客户名称 | 长度=2（最小） | 客户名称输入"张" |
| 客户名称 | 长度=100（最大） | 客户名称输入100个字符 |
| 客户名称 | 长度=1（最小-1） | 客户名称输入1个字符，预期失败 |
| 客户名称 | 长度=101（最大+1） | 客户名称输入101个字符，预期失败 |
| 商机金额 | 金额=0（边界） | 金额输入0，预期失败 |
| 商机金额 | 金额=0.01（最小） | 金额输入0.01，预期成功 |
| 商机金额 | 金额=-1（非法） | 金额输入-1，预期失败 |

#### 2. 等价类划分法
- **适用场景**：输入数据分类、权限等级、状态分类
- **设计要点**：每个等价类对应独立用例，不可合并
- **CRM接口等价类示例**：

| 等价类 | 测试用例 |
|--------|----------|
| 有效等价类-普通客户 | 创建客户，level="普通" |
| 有效等价类-VIP客户 | 创建客户，level="VIP" |
| 无效等价类-空字符串 | 创建客户，name="" |
| 无效等价类-特殊字符 | 创建客户，name="<>script" |
| 无效等价类-SQL注入 | 创建客户，name="'; DROP TABLE users;--" |

#### 3. 错误猜测法
- **适用场景**：异常操作、非法输入、系统异常
- **设计要点**：基于经验猜测可能出现的错误
- **CRM接口错误猜测示例**：

| 错误场景 | 测试用例 |
|----------|----------|
| 未授权访问 | 不登录直接访问需要认证的接口 |
| 重复提交 | 短时间内多次发送相同请求 |
| 并发操作 | 多个请求同时修改同一数据 |
| 网络异常 | 请求超时或中断 |

#### 4. 安全测试
- **适用场景**：所有涉及用户输入的接口
- **测试要点**：
  - SQL注入测试
  - XSS攻击测试
  - CSRF跨站请求伪造测试
  - 敏感信息泄露测试
  - 暴力破解防护测试

| 安全测试类型 | 测试用例 |
|--------------|----------|
| SQL注入 | 用户名输入`' OR '1'='1` |
| XSS攻击 | 客户名称输入`<script>alert('xss')</script>` |
| 暴力破解 | 连续5次错误密码登录 |

## 三、业务流程测试

### 功能说明
将多个API串联起来，模拟完整的业务流程，验证端到端业务逻辑。

### 适用场景
- 端到端测试
- 业务流程验证
- 集成测试

### 业务流程定义规范

#### 变量提取与传递
- 从响应中提取关键数据作为上下文变量
- 在后续请求中使用`{{变量名}}`进行替换
- 支持JSONPath表达式提取数据

#### 流程步骤定义
```yaml
business_flows:
  - name: "客户跟进全流程"
    flow_id: "FLOW-001"
    description: "从登录到客户创建、跟进记录、查看详情的完整流程"
    steps:
      - name: "Step1-用户登录"
        step_id: "STEP-001"
        api: "/api/users/login"
        method: POST
        body: {"username": "admin", "password": "123456"}
        extract:
          user_id: "$.user.id"
          username: "$.user.username"
      
      - name: "Step2-新增客户"
        step_id: "STEP-002"
        api: "/api/customers"
        method: POST
        body: {"name": "测试客户有限公司", "phone": "13800138000", "industry": "互联网"}
        extract:
          customer_id: "$.customer.id"
          customer_name: "$.customer.name"
      
      - name: "Step3-新增跟进记录"
        step_id: "STEP-003"
        api: "/api/followups"
        method: POST
        body: {"customer_id": "{{customer_id}}", "type": "电话", "content": "首次客户跟进沟通"}
        extract:
          followup_id: "$.id"
      
      - name: "Step4-查看客户详情"
        step_id: "STEP-004"
        api: "/api/customers/{{customer_id}}"
        method: GET
        assertions:
          - "customer_id == {{customer_id}}"
          - "customer_name == '测试客户有限公司'"
```

## 工作流程

### Step 1: 分析代码结构
1. 解析路由定义
2. 提取请求方法和参数
3. 分析请求/响应结构
4. 识别数据模型

### Step 2: 生成swagger.yaml
1. 定义基础信息（swagger版本、info、host、basePath）
2. 定义数据模型（definitions）
3. 定义接口路径（paths），包含：
   - summary：接口名称
   - description：接口描述和业务规则
   - parameters：参数定义（包含minLength、maxLength、pattern等约束）
   - responses：响应定义（必须包含示例）

### Step 3: 生成测试用例
1. 生成Mock测试数据
2. 生成单接口测试用例（边界值、等价类、错误推断、安全测试）
3. 生成业务流程测试用例

### Step 4: 生成swagger.html
使用以下模板生成HTML文件：

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>API 文档</title>
  <link rel="stylesheet" href="https://cdn.bootcdn.net/ajax/libs/swagger-ui/4.15.5/swagger-ui.css">
</head>
<body>
  <div id="swagger-ui"></div>
  <script src="https://cdn.bootcdn.net/ajax/libs/swagger-ui/4.15.5/swagger-ui-bundle.js"></script>
  <script src="https://cdn.bootcdn.net/ajax/libs/js-yaml/4.1.0/js-yaml.min.js"></script>
  <script>
    window.onload = function() {
      fetch('swagger.yaml')
        .then(response => response.text())
        .then(yamlContent => {
          const spec = jsyaml.load(yamlContent);
          window.ui = SwaggerUIBundle({
            spec: spec,
            dom_id: '#swagger-ui',
            presets: [SwaggerUIBundle.presets.apis],
            layout: "BaseLayout"
          });
        });
    };
  </script>
</body>
</html>
```

## 测试结果分析规范

### 缺陷严重程度定义

| 严重程度 | 定义 | 示例 |
|-----------|------|------|
| Blocker | 系统无法运行 | 登录功能完全不可用 |
| Critical | 核心功能严重受损 | 客户数据无法保存 |
| Major | 功能未按预期工作 | 边界值校验失效 |
| Minor | 界面/体验问题 | 提示信息不友好 |
| Trivial | 微小问题 | 错别字、排版问题 |

### 缺陷修复验证
- 缺陷修复后必须重新执行相关测试用例
- 回归测试必须覆盖缺陷相关的核心路径
- 缺陷单关闭前需确认所有关联用例通过
