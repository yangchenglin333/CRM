# 测试用例生成规则

## 必须遵守的原则

### 执行原则（最高优先级）

**必须严格按照本规则执行，禁止参照已有用例结构或数量。**

- ❌ **禁止**：参照已有用例文件（如某模块的CSV）的结构、数量、风格来生成用例
- ❌ **禁止**：以"参考用例有N个，所以本模块也生成N个"为由控制用例数量
- ✅ **正确**：以需求文档为唯一依据，按本规则中的覆盖率要求（正向100%、反向80%）、测试方法应用要求、等价类/边界值原则逐项执行
- ✅ **正确**：生成前先做需求拆解与测试设计（功能点清单、等价类、边界值、状态迁移等），再按设计生成用例
- ✅ **正确**：生成后按「用例自检标准」逐条自检，漏测功能点必须补足

### 强制执行流程（不得跳过）

**生成测试用例时，必须按以下顺序执行，不得跳过任一步骤：**

| 步骤 | 动作 | 产出 | 说明 |
|------|------|------|------|
| **Step 0** | 需求文档格式转换 | 原始需求文档(.docx/.doc) → Markdown格式(.md) | **输入为Word文档时必须执行**，使用 `其他/文件转换/Word转Markdown.py` 工具转换 |
| **Step 1** | 需求拆解 | 功能点清单、等价类、边界值、状态迁移 | 生成前必须先完成，作为用例设计依据 |
| **Step 2** | 用例生成 | 按清单逐项生成用例 | **大型需求文档必须分批生成**，禁止一次性写入 |
| **Step 3** | 自检 | 按「用例自检标准」逐条核对，标注漏测项 | 生成完成后必须执行 |
| **Step 4** | 补足 | 补充漏测用例 | 自检发现漏测后必须补足，方可输出最终 CSV |

**禁止行为**：不得跳过 Step 0 直接读取Word文档、不得跳过 Step 1 直接生成、不得跳过 Step 3 自检、不得在发现漏测后不补足即结束。

**Step 0 转换说明**：
- **工具位置**：`其他/文件转换/Word转Markdown.py`
- **转换命令**：`python3 Word转Markdown.py <输入docx文件> -o <输出md文件>`
- **批量转换**：`python3 Word转Markdown.py <输入目录> -o <输出目录> -b`
- **原因**：Markdown格式更易于程序解析（正则匹配、字符串处理），版本控制友好，可清晰对比变更历史

### 功能点清单强制产出（Step 1 细化）

**Step 1 必须产出结构化功能点清单，作为后续生成、自检、补足的唯一依据。**

#### 清单格式（强制）

| 功能点ID | 需求章节 | 功能点描述 | 测试要点 | 等价类/边界值 | 覆盖状态 |
|----------|----------|------------|----------|---------------|----------|
| F001 | 3.7.1 | 提前立项-提交 | 必填、开工内容校验 | 建设类必填/结算类非必填、500字符边界 | 待覆盖 |
| F002 | 3.7.1 | 提前立项-销售经理审批 | 同意/拒绝/退回 | - | 待覆盖 |

- **功能点ID**：F001、F002... 按需求文档顺序编号
- **需求章节**：需求文档中的章节号（如 3.7.1、3.7.2、3.8.1）
- **功能点描述**：简洁描述该功能点
- **测试要点**：需验证的具体点
- **等价类/边界值**：该功能点涉及的等价类、边界值
- **覆盖状态**：待覆盖 → 已覆盖（填对应 TC-xxx）

#### 清单提取规则

- **必须**逐章逐节阅读需求文档，从 3.7.x、3.8.x 等功能说明章节提取
- **每个**功能模块下的**每个**可测试点（提交、审批、校验、查询、导出、权限等）均须列入清单
- **暂不做的**功能（如文档标注「暂时不做」）可列清单但标注「本期不测」

### 需求-用例映射表（强制）

**生成用例时，必须建立需求-用例映射，确保可追溯。**

| 功能点ID | 对应用例标识 | 备注 |
|----------|--------------|------|
| F001 | TC-001, TC-002 | 提前立项提交 |
| F002 | TC-007 | 销售经理审批 |

- 每个功能点至少有一个对应用例
- 一个用例可覆盖多个功能点，但每个功能点必须有至少一个用例

### 自检清单打勾（Step 3 细化）

**Step 3 自检必须按功能点清单逐条打勾，不得凭主观判断。**

#### 打勾流程

1. 读取功能点清单，逐行检查
2. 对每个功能点：在已生成用例中查找是否有覆盖该功能点的用例
3. 若有 → 在清单「覆盖状态」列填「已覆盖 TC-xxx」
4. 若无 → 在「覆盖状态」列填「漏测」，并加入**漏测清单**

#### 漏测清单（强制产出）

| 功能点ID | 需求章节 | 功能点描述 | 漏测原因 |
|----------|----------|------------|----------|
| F015 | 3.7.4 | 全景视图-变更记录 | 未生成对应用例 |

- 自检完成后，漏测清单**必须非空则进入 Step 4 补足**
- **漏测清单未清空前，禁止输出最终 CSV**

### 补足基于漏测清单（Step 4 细化）

**Step 4 补足必须针对漏测清单中的每一项，逐项补充用例。**

1. 按漏测清单顺序，为每项漏测功能点生成至少 1 个用例
2. 每补足一项，在功能点清单中更新「覆盖状态」为「已覆盖 TC-xxx」
3. 从漏测清单中移除该项
4. 漏测清单清空后，方可输出最终 CSV

### 验收标准（强制）

**以下条件全部满足后，方可输出最终 CSV：**

- [ ] 功能点清单已完整产出
- [ ] 功能点清单中每个功能点「覆盖状态」均为「已覆盖」
- [ ] 漏测清单为空
- [ ] 需求-用例映射表已建立

**违反上述任一条件，禁止输出。** 必须回到对应步骤补全后再输出。

### CSV与JSON同步更新规范（强制）

**每次生成或修改测试用例时，必须同时更新CSV和JSON两种格式：**

1. **同步触发条件**：以下任一情况发生时，必须同时更新CSV和JSON文件：
   - 新增测试用例时
   - 修改测试用例时
   - 删除测试用例时
   - 用例ID重新编号时

2. **版本更新要求**：
   - **每次更新必须生成新版本文件**：不能覆盖之前的用例文件
   - **版本号递增**：v1.0 → v1.1 → v2.0
   - **保留历史文件**：所有历史版本的用例文件都必须保留
   - **命名格式**：`YYYYMMDD_{系统名}_{类型}_用例_{版本}.csv/json`

3. **CSV格式要求**（必须包含的列）：
   | 列名 | 说明 | 示例 |
   |------|------|------|
   | 用例ID | 唯一标识 | TC-001 |
   | 关联需求 | 需求ID | R001,R002 |
   | 测试场景 | 场景描述 | 登录-正常登录成功 |
   | 前置条件 | 前提条件 | 系统已启动 |
   | 步骤描述 | 操作步骤 | 1.输入用户名\n2.输入密码 |
   | 输入 | 测试数据 | 用户名：admin |
   | 期望输出 | 预期结果 | 登录成功 |
   | 优先级 | P0/P1/P2 | P0 |

4. **JSON格式要求**（必须包含的字段）：
   ```json
   {
     "test_case_id": "TC-001",
     "name": "用户登录-正常登录成功",
     "priority": "P0",
     "precondition": "系统已启动",
     "steps": [
       {
         "step_name": "步骤1",
         "action": "open",
         "target": "/login",
         "value": "",
         "expected": "登录页面正常显示"
       }
     ]
   }
   ```

5. **同步字段对应关系**：
   | CSV列名 | JSON字段 | 说明 |
   |----------|----------|------|
   | 用例ID | test_case_id | 用例唯一标识 |
   | 测试场景 | name | 用例名称 |
   | 优先级 | priority | P0/P1/P2 |
   | 前置条件 | precondition | 前置条件 |
   | 步骤描述 | steps[].action/target/value | 操作步骤 |
   | 期望输出 | steps[].expected | 预期结果 |

6. **更新后自检**：
   - [ ] CSV和JSON文件同步更新
   - [ ] 生成了新版本的用例文件（版本号递增）
   - [ ] 历史版本的用例文件已保留
   - [ ] 两个文件的用例数量一致
   - [ ] 两个文件的用例ID一致

### 分批生成触发条件（强制）

**满足以下任一条件时，必须采用分批生成策略，禁止一次性 create_test_case_data() 写入：**

- 需求文档行数 **> 500 行**
- 需求文档大小 **> 500KB**
- 预估用例数 **> 50 个**
- 功能模块数 **≥ 5 个**（如 3.7.1、3.7.2、3.7.3…）

**分批生成方式**：
- 方式一：使用 `search_replace` 工具按功能模块分批追加（每批 5-15 个用例）
- 方式二：使用 `其他/大文件处理方案脚本.py`（预估用例 > 100 时推荐）

**❌ 禁止**：一次性在 create_test_case_data() 中写入全部用例后立刻生成 CSV 并删除脚本。

### 需求来源原则（强制）

**生成测试用例时必须依据原始需求文档（Markdown格式）和本规则，禁止依据需求提炼分析生成。**

- ✅ **正确**：将Word文档(.docx)先转换为Markdown格式(.md)，直接阅读.md文件，逐章逐节提取功能点、业务流程、业务规则，按本规则生成完整测试用例
- ✅ **正确**：以原始需求文档中的功能描述、界面要素、数据字段、校验规则为唯一输入来源
- ❌ **禁止**：直接读取原始Word文档(.docx)进行用例生成（程序解析困难，容易遗漏需求）
- ❌ **禁止**：以「需求提炼分析」「需求理解输出」「测试关键点分析」等衍生文档作为生成依据
- ❌ **禁止**：仅按需求提炼分析的摘要或清单生成用例，导致漏测原始需求中的详细功能点

**原因说明**：
1. 测试周期评估按**已完成的测试用例功能**来评估执行时间；若用例基于提炼分析而非原始需求，会导致功能覆盖不全、用例数量偏少，从而低估测试周期
2. Markdown格式更易于程序解析（正则匹配、字符串处理），版本控制友好，可清晰对比变更历史
3. Word文档需要额外解析库（python-docx），且格式复杂，容易出现解析错误

## 核心原则

1. **充分理解需求文档**（前提条件）：生成测试用例前，必须充分理解需求文档的全部内容，包括功能描述、业务流程、业务规则、界面要求、数据要求等。

2. **严格遵循需求文档**（最重要）：生成用例时必须严格遵循需求文档，严禁出现需求文档中没有提到的功能。
   - ✅ **正确**：需求文档明确提到"新增配置"功能 → 生成"新增配置"的测试用例
   - ✅ **正确**：需求文档明确提到"失效配置"功能 → 生成"失效配置"的测试用例
   - ❌ **错误**：需求文档只提到"新增、失效配置"，未提到"编辑配置" → 严禁生成"编辑配置"的测试用例
   - ❌ **错误**：需求文档未明确描述的功能 → 严禁自行推测并生成测试用例

3. **步骤与结果一一对应**（核心原则）：每个测试步骤必须有且仅有一个对应的预期结果，步骤和结果必须严格一一对应。
   - **数量一致**：测试步骤数量 = 预期结果数量
   - **顺序一致**：预期结果的顺序与测试步骤的顺序完全一致
   - **即时验证**：每个步骤执行后，应立即验证对应的预期结果
   - ✅ **正确**：步骤1"点击查询按钮" → 预期结果1"可以点击，展示所有查询的结果"
   - ❌ **错误**：步骤1"点击查询按钮" → 预期结果"系统能够查询数据"（没有对应关系）

4. **步骤独立可执行**：每个测试步骤必须独立、具体、可执行，不能合并多个操作。每个步骤应描述具体的操作动作，包含操作对象、操作方式和输入数据。

5. **测试步骤清晰度要求**（新增，重要）：测试步骤必须清晰明确，便于UI自动化准确识别和执行。
   - **模块定位**：每个步骤必须明确指出操作所在的模块位置（如：POM-项目管理-项目列表页）
   - **具体动作**：使用明确的动词描述操作（如：点击、输入、选择、勾选、拖拽等）
   - **元素定位**：明确指出操作的具体元素（如：点击"新增"按钮、在"项目名称"输入框中输入）
   - **数据明确**：步骤中包含具体的测试数据，不使用模糊描述
   - **错误定位**：预期结果中明确指出错误发生的位置和具体信息
   - ✅ **正确示例**：
     - 步骤1：在"POM-项目管理-项目列表页"中，点击"新增项目"按钮
     - 步骤2：在弹出的新增项目表单中，在"项目名称"输入框中输入"测试项目A"
     - 步骤3：在"项目类型"下拉框中选择"新建项目"
     - 步骤4：点击"保存"按钮
     - 预期结果4：系统提示"保存成功"，项目列表中新增一条记录，项目名称为"测试项目A"
   - ❌ **错误示例**：
     - 步骤1：新增项目（缺少模块位置和具体动作）
     - 步骤2：输入项目信息（缺少具体字段和数据）
     - 步骤3：保存（缺少按钮名称）

6. **UI自动化识别优化**（新增，重要）：测试步骤设计要便于MCP模式和传统Playwright准确识别和执行。
   - **元素定位策略**：优先使用稳定的定位方式（如：data-testid、aria-label、明确的文本内容）
   - **操作明确性**：每个操作步骤只包含一个明确的动作，不合并多个操作
   - **等待机制**：在需要等待的步骤中明确等待条件（如：等待页面加载完成、等待弹窗出现）
   - **错误处理**：在预期结果中明确错误提示的具体位置和内容（如：在"项目名称"输入框下方显示红色错误提示"项目名称不能为空"）
   - **状态验证**：每个操作后验证相应的状态变化（如：按钮状态、页面跳转、数据更新）

7. **覆盖率要求**（重要）：
   - **正向用例覆盖率**：正向用例务必达到百分之百（100%）覆盖
   - **反向用例覆盖率**：反向用例的覆盖度不得低于百分之八十（80%）。反向用例主要指**场景类用例**（因果图、场景法、判定表），此类场景用例数量应占正向用例总数的**20%**。
   - **测试关键点 100% 覆盖**（强制）：当使用「需求文档 + 测试关键点分析」生成用例时，**必须达到 100% 覆盖**所有测试场景。

8. **测试方法综合应用**（重要）：必须充分利用各种测试方法（边界值分析、等价类划分、错误猜测、因果图、场景法、判定表等），确保测试覆盖范围。
   - **临界值与等价类用例比例**：临界值分析、等价类划分生成的用例数量，合计占正向用例总数的**20%**。例如：正向用例生成100条，则临界值和等价类相关用例共生成20条。
   - **因果图、场景法、判定表全覆盖**（强制）：涉及多条件组合、业务流程、业务规则的需求，必须对因果图、场景法、判定表进行**全覆盖**，确保流程的完整性和准确性。

9. **前置条件单一场景原则**（重要）：每条用例的前置条件仅需包含一种场景/临界值，确保用例的独立性和可执行性。
   - ✅ **正确**：`该项目为已签约非框架项目，存在待完成的经营管理里程碑，项目状态为在建，经营管理里程碑当前计划完成年月较基版延迟2个月`
   - ❌ **错误**：`项目可能为框架或非框架项目，延迟1-3个月`（混合了多种场景）

10. **测试名称规范**（重要）：测试名称必须包含完整的测试信息，格式为：**模块-测试点-测试条件-测试结果（可选）**
    - ✅ **正确**：`全景视图-进度状态取值验证-已签约非框架-经营里程碑有待完成-在建-延迟2个月`
    - ✅ **正确**：`项目管理-新增项目-管理员权限-必填字段完整-保存成功`
    - ❌ **错误**：`测试新增功能`（缺少模块、测试点、测试条件）

11. **反向场景全面性原则**（重要）：反向场景必须全面覆盖所有边界情况，对于范围型条件，必须拆分为独立用例。
    - 需求描述：`轻度延迟：在建/竣工项目，订单当前赢率≥50%，当前预签年月较提前立项时延迟1-3个月`
    - 必须生成的用例：
      - TC-001：`轻度延迟-在建项目-赢率50%-延迟小于1个月（如0.5个月）` → 预期：不属于轻度延迟
      - TC-002：`轻度延迟-在建项目-赢率50%-延迟等于1个月` → 预期：属于轻度延迟
      - TC-003：`轻度延迟-在建项目-赢率50%-延迟等于2个月` → 预期：属于轻度延迟
      - TC-004：`轻度延迟-在建项目-赢率50%-延迟等于3个月` → 预期：属于轻度延迟
      - TC-005：`轻度延迟-在建项目-赢率50%-延迟大于3个月（如4个月）` → 预期：不属于轻度延迟

12. **用例质量保证**：
    - **可追溯性**：测试用例与需求对应，便于追溯
    - **独立性**：每个测试用例应独立，不依赖其他用例的执行结果
    - **可维护性**：用例结构清晰，便于维护和更新
    - **用例自检**：生成完成后必须进行至少一次自检

13. **生成后删除测试脚本**（重要）：每次生成完测试用例（CSV 生成完成后），必须将本次用于生成该模块测试用例的**专用脚本删除，不保留**。

14. **生成后校验**（重要）：CSV 生成完成后，必须确保文件命名与显示顺序与历史文件统一。

## 测试方法应用要求（详细版）

### 1. 边界值分析法（细化）

- **适用场景**：
  - 数值输入（如金额、数量、比例等）
  - 日期范围（如开始日期、结束日期）
  - 数量限制（如文件大小、字符长度）
  - Cursor操作边界（如元素尺寸边界、滚动边界）

- **设计要点**：
  - 覆盖六点边界值：**最小值-1、最小值、最小值+1、最大值-1、最大值、最大值+1**
  - 结合Cursor操作验证边界触发效果
  - 对于范围型条件（如延迟1-3个月），必须拆分为独立用例：
    - 小于最小值（如延迟<1个月）
    - 等于最小值（如延迟=1个月）
    - 范围内中间值（如延迟=2个月）
    - 等于最大值（如延迟=3个月）
    - 大于最大值（如延迟>3个月，如4个月）

- **Cursor融合示例**：
  - 步骤："用Cursor点击输入框，输入101个字符（'A'重复101次）"

- **完整示例**：

    - TC-002：输入1个字符 → 输入成功，保存正常
    - TC-003：输入2个字符 → 输入成功，保存正常
    - TC-004：输入99个字符 → 输入成功，保存正常

### 2. 等价类划分法（细化）

- **适用场景**：

  - 状态分类（如项目状态：在建、竣工、已签约等）

- **设计要点**：
  - **有效等价类**：覆盖所有合法输入/操作，每个有效等价类对应独立用例
  - **无效等价类**：覆盖所有非法、异常场景，每个无效等价类对应独立用例
  - 每个等价类必须有独立的测试用例，不可合并

- **Cursor融合示例**：
  - 步骤："用Cursor点击用户名输入框，输入2个字符'AB'，左击页面其他区域（失焦）"
  - 预期结果："输入框下方显示'用户名长度需3-20字符'提示，输入框边框变为红色"

- **完整示例**：
  - 用户名输入（3-20字符）：
    - 有效等价类：
      - TC-001：输入3个字符 → 验证通过
      - TC-002：输入10个字符（中间值）→ 验证通过
      - TC-003：输入20个字符 → 验证通过
    - 无效等价类：

      - TC-005：输入21个字符 → 提示"用户名长度需3-20字符"
      - TC-006：输入特殊字符"@#$" → 提示"用户名只能包含字母和数字"
      - TC-007：输入空值 → 提示"用户名不能为空"

### 3. 错误猜测法（细化）

- **适用场景**：
  - 异常操作（如未登录访问、权限不足）

- **测试要点**：
  - 基于经验和直觉，猜测可能出现的错误
  - 测试异常输入、非法操作、系统边界情况
  - 测试用户可能误操作的情况

- **Cursor相关异常场景（新增）**：
  - Cursor快速切换操作（点击后立即悬停）

- **完整示例**：

  - 页面未完全加载时Cursor点击按钮

### 4. 因果图法（细化）

- **适用场景**：
  - 多个输入条件组合
  - 复杂业务逻辑判断
  - 条件之间存在依赖关系

- **全覆盖要求**（强制）：涉及多条件组合、条件间存在因果/依赖关系的需求，必须对因果图进行**全覆盖**，为每个条件组合生成对应用例，不得遗漏任一组合，以确保流程的完整性和准确性。

- **测试要点**：
  - 分析输入条件之间的逻辑关系（与、或、非）
  - 设计测试用例覆盖**所有**条件组合（全覆盖）
  - 考虑条件之间的依赖关系和约束关系

- **Cursor融合要点**：
  - 在因果图中加入Cursor操作状态作为条件
  - 考虑Cursor操作顺序对结果的影响

- **完整示例**：
  - 涉及法律内容（是/否）与审批流程的因果关系：
    - 条件1：涉及法律内容=是
    - 条件2：金额>10万
    - 结果：需要法务审批
  - 组合用例：
    - TC-001：涉及法律内容=是，金额>10万 → 需要法务审批
    - TC-002：涉及法律内容=是，金额≤10万 → 需要法务审批

    - TC-004：涉及法律内容=否，金额≤10万 → 不需要法务审批

### 5. 场景法（细化）

- **适用场景**：
  - 业务流程测试
  - 用户使用场景模拟
  - 端到端测试

- **全覆盖要求**（强制）：涉及业务流程的需求，必须对场景法进行**全覆盖**，覆盖正常流程、异常流程、备选流程的**所有**分支，不得遗漏任一流程路径，以确保流程的完整性和准确性。

- **测试要点**：
  - 基于真实业务场景设计测试用例
  - 覆盖**所有**正常流程、异常流程、备选流程（全覆盖）
  - 模拟用户真实使用路径

- **Cursor融合要点**：
  - 完整业务流程需包含Cursor操作全路径
  - 每个环节明确Cursor操作细节
  - 覆盖正常流程和异常中断流程（如中途用Cursor关闭页面）

- **Cursor融合示例**：
  - 完整流程："登录→导航→新增→提交→查询"
  - 每个环节明确Cursor操作：

    - 步骤3："用Cursor点击'新增项目'按钮"
    - ...

- **异常中断流程示例**：

  - 预期结果："系统弹出'是否保存草稿'确认框"

### 6. 判定表法（细化）

- **适用场景**：
  - 多条件组合判断
  - 复杂业务规则
  - 权限矩阵验证

- **全覆盖要求**（强制）：涉及多条件判断、业务规则、权限矩阵的需求，必须对判定表进行**全覆盖**，为判定表中的**每一列（每条规则）**生成对应用例，不得遗漏任一规则，以确保流程的完整性和准确性。

  - 设计测试用例覆盖**判定表全部规则**（全覆盖）
  - 确保规则覆盖完整

- **Cursor融合要点**：

  - 条件项可包含"Cursor悬停状态"、"Cursor点击状态"等
  - 确保组合覆盖无遗漏

## 用例自检标准（强制要求）

生成测试用例后，必须按照以下标准进行自检，确保用例质量符合要求。

### 一、需求一致性（必达标，核心优先级）

#### 1.1 无超需求用例（强制要求）

**检查要点**：用例功能/场景必须在需求文档有明确描述，禁止推测未提及功能。

**规则说明**：
- ✅ **正确**：需求文档明确提到"新增配置"功能 → 生成"新增配置"的测试用例
- ✅ **正确**：需求文档明确提到"失效配置"功能 → 生成"失效配置"的测试用例
- ❌ **错误**：需求文档只提到"新增、失效配置"，未提到"编辑配置" → 严禁生成"编辑配置"的测试用例
- ❌ **错误**：需求文档未明确描述的功能 → 严禁自行推测并生成测试用例

**示例**：
- 需求仅提"新增、失效配置"，若生成"编辑配置"用例，判定违规

**整改要求**：直接删除超需求用例，不额外新增需求外场景。

#### 1.2 需求全覆盖（强制要求）

**检查要点**：
- **正向功能点**：100%覆盖
- **反向异常场景**：覆盖≥25%

**规则说明**：
- 需求文档中提到的每个正向功能点，都必须有对应的正向测试用例
- 对于每个正向功能点，需要补充足够的反向用例（异常场景、边界情况、错误输入等）
- 反向用例覆盖率必须达到25%以上

**示例**：
- 需求提"管理员新建配置"（正向），需对应TC-001（正向用例）
- 同时补充"普通用户新建配置（权限不足）"、"配置名称为空（非法输入）"等反向用例，确保反向覆盖达标

**整改要求**：
- 缺正向用例：补全对应功能点用例
- 反向覆盖率不足：补充异常场景用例，确保反向覆盖达标

### 二、用例质量（必达标，保障可用性）

#### 2.1 无重复用例（强制要求）

**检查要点**：同模块、同功能、同场景的用例不可重复（核心步骤一致即判定重复）。

**规则说明**：
- 如果两个用例的测试场景、核心步骤完全一致，仅数据不同，判定为重复用例
- 如果两个用例的测试场景、核心步骤基本一致，仅细微差异，判定为重复用例
- 重复用例必须合并或删除

**示例**：
- TC-002与TC-005均为"管理员新建配置（名称合规、比例50%）"，步骤与场景完全一致，判定重复

**整改要求**：
- 保留1个用例，删除其余
- 若仅数据不同（如比例30%/70%），可合并为1个用例（步骤注明数据范围，结果对应不同数据反馈）

#### 2.2 步骤与结果对应完整性（补充）

**检查要点**：每个测试步骤都必须有对应的预期结果，步骤和结果数量、顺序必须一致。

**规则说明**：
- 检查每个用例的步骤数量与预期结果数量是否一致
- 检查每个步骤是否都有对应的预期结果
- 检查预期结果的顺序是否与步骤顺序一致
- 检查预期结果是否具体、可验证

**整改要求**：
- 补充缺失的预期结果
- 调整预期结果的顺序，确保与步骤顺序一致
- 细化预期结果描述，使其具体可验证

#### 2.3 用例可执行性检查（补充）

**检查要点**：每个测试用例是否可以直接执行，无需额外说明或推测。

**规则说明**：
- 检查步骤描述是否具体明确，包含操作对象、操作方式、输入数据
- 检查前置条件是否完整，是否包含所有必要的环境、数据、权限要求
- 检查步骤描述中是否包含具体的测试数据（虽然CSV的"输入"字段需要提取数据，但步骤描述中也应包含具体数据）

**整改要求**：
- 补充缺失的操作细节
- 完善前置条件描述

### 三、测试方法覆盖（核心功能必达标，保障全面性）

#### 3.1 核心功能覆盖（强制要求）

**检查要点**：新增、查询、失效等核心功能，至少覆盖2种测试方法。

**规则说明**：
- 对于核心功能（如新增、查询、失效、删除等），必须应用至少2种不同的测试方法
- 测试方法包括：边界值分析、等价类划分、错误猜测、因果图、场景法、判定表等
- 仅用1种方法判定覆盖不足

**示例**：
- "新建配置"（核心功能）需同时用"等价类划分"（合法名称/非法名称）和"边界值分析"（比例0%、100%、1%、99%）
- 仅用1种方法判定覆盖不足

**整改要求**：补充对应测试方法的用例，确保核心功能方法覆盖充足。

#### 3.2 因果图、场景法、判定表全覆盖（强制要求）

**检查要点**：涉及多条件组合、业务流程、业务规则的需求，因果图、场景法、判定表是否全覆盖。

**规则说明**：
- **因果图**：多条件组合、条件间存在因果/依赖关系时，必须为每个条件组合生成用例，不得遗漏
- **场景法**：业务流程需求必须覆盖正常、异常、备选流程的所有分支，不得遗漏任一路径
- **判定表**：业务规则、权限矩阵等必须为判定表每一列（每条规则）生成用例，不得遗漏
- 未达到全覆盖判定违规

**整改要求**：补充遗漏的条件组合、流程分支、判定规则用例，确保流程完整性和准确性。

#### 3.3 反向用例覆盖（强制要求）

**检查要点**：反向场景需应用错误猜测法/无效等价类。

**规则说明**：
- 反向用例必须明确应用错误猜测法或无效等价类划分法
- **错误猜测法**：基于经验和直觉，猜测可能出现的错误（如权限不足、网络异常、数据为空等）
- **无效等价类**：不符合要求的输入数据（如格式错误、范围超限、必填项为空等）
- 未应用对应方法判定违规

**示例**：
- 反向用例"无权限用户操作失效配置"用错误猜测法
- "配置生效日期早于当前日期"用无效等价类
- 未应用对应方法判定违规

**整改要求**：
- 补充反向场景用例
- 或调整用例适配对应测试方法

#### 3.4 临界值与等价类用例比例检查（补充）

**检查要点**：临界值分析、等价类划分生成的用例数量，合计是否约为正向用例总数的20%。

**规则说明**：例如正向用例100条，则临界值+等价类相关用例共约20条。比例过高或过低需调整。

#### 3.5 异常场景完整性检查（补充）

**检查要点**：检查是否覆盖了所有可能的异常场景。

**规则说明**：
- 检查是否覆盖了系统异常场景（如网络异常、服务异常、数据库异常等）
- 检查是否覆盖了业务异常场景（如数据冲突、业务规则违反等）
- 检查是否覆盖了用户操作异常场景（如误操作、非法操作等）
- 检查是否覆盖了数据异常场景（如数据为空、数据格式错误、数据范围超限等）

**整改要求**：
- 补充缺失的异常场景用例
- 确保异常场景覆盖全面

### 四、用例规范性检查（新增，必达标）

#### 4.1 前置条件单一场景检查（强制要求）

**检查要点**：每条用例的前置条件是否仅包含一种场景/临界值。

**规则说明**：
- 前置条件应明确描述单一的测试场景，不能混合多种不同场景
- 前置条件应包含所有必要的条件组合，但每个条件值必须是确定的
- 禁止在前置条件中使用"或"、"可能"、"范围"等模糊描述

**正确示例**：
- ✅ `该项目为已签约非框架项目，存在待完成的经营管理里程碑，项目状态为在建，经营管理里程碑当前计划完成年月较基版延迟2个月`
- ✅ `用户角色为管理员，项目状态为竣工，订单赢率为60%`

**错误示例**：
- ❌ `项目可能为框架或非框架项目`（混合了多种场景）
- ❌ `延迟1-3个月`（范围型描述，应拆分为独立用例）
- ❌ `用户角色为管理员或普通用户`（混合了多种角色）

**整改要求**：
- 将混合场景的用例拆分为多个独立用例
- 每个用例的前置条件只描述一种确定的场景

#### 4.2 测试名称规范检查（强制要求）

**检查要点**：测试名称是否符合"模块-测试点-测试条件-测试结果"格式。

**规则说明**：
- 测试名称必须包含：模块、测试点、测试条件
- 测试结果为可选项，当结果明确时可省略
- 多个测试条件之间使用"-"连接
- 测试名称应能清晰表达测试场景，便于理解和追溯

**正确示例**：
- ✅ `全景视图-进度状态取值验证-已签约非框架-经营里程碑有待完成-在建-延迟2个月`
- ✅ `项目管理-新增项目-管理员权限-必填字段完整-保存成功`
- ✅ `订单管理-赢率验证-在建项目-赢率50%-轻度延迟1个月`
- ✅ `用户管理-权限验证-普通用户-访问管理员功能-拒绝访问`

**错误示例**：
- ❌ `测试新增功能`（缺少模块、测试点、测试条件）
- ❌ `项目查询测试`（缺少测试条件）
- ❌ `TC-001`（仅有编号，无描述信息）
- ❌ `验证延迟场景`（缺少具体条件）

**整改要求**：
- 补充缺失的模块、测试点、测试条件信息
- 按照规范格式重新命名测试用例

#### 4.3 反向场景全面性检查（强制要求）

**检查要点**：反向场景是否全面覆盖所有边界情况，范围型条件是否拆分为独立用例。

**规则说明**：
- 当需求文档描述的条件包含范围（如"延迟1-3个月"、"赢率50%-80%"），必须生成以下独立用例：
  - 小于范围最小值的场景
  - 等于范围最小值的场景
  - 范围内中间值的场景（至少1个）
  - 等于范围最大值的场景
  - 大于范围最大值的场景
- 每个边界值必须有独立的测试用例，不能合并

**检查示例**：
- 需求描述：`轻度延迟：在建/竣工项目，订单当前赢率≥50%，当前预签年月较提前立项时延迟1-3个月`
- 必须检查是否存在以下用例：
  - ✅ 延迟小于1个月（如0.5个月）的用例
  - ✅ 延迟等于1个月的用例
  - ✅ 延迟等于2个月的用例（中间值）
  - ✅ 延迟等于3个月的用例
  - ✅ 延迟大于3个月（如4个月）的用例

**整改要求**：
- 补充缺失的边界值用例
- 将范围型用例拆分为独立的边界值用例
- 确保每个边界点都有对应的测试用例

### 自检流程

生成测试用例后，必须按照以下流程进行自检：

1. **第一步：需求一致性检查**
   - 逐一核对每个用例是否在需求文档中有对应描述
   - 检查正向功能点是否100%覆盖
   - 检查反向用例覆盖率是否≥80%，场景类（因果图、场景法、判定表）用例占比是否约20%
   - 删除超需求用例
   - 补充缺失的正向/反向用例

2. **第二步：用例质量检查**
   - 识别并删除重复用例
   - 合并仅数据不同的用例
   - 检查步骤与结果对应完整性
   - 检查用例可执行性

3. **第三步：测试方法覆盖检查**
   - 检查核心功能是否应用了至少2种测试方法
   - 检查反向用例是否应用了错误猜测法/无效等价类
   - 检查异常场景是否覆盖完整
   - 检查临界值+等价类用例占比是否约为正向用例的20%
   - 检查因果图、场景法、判定表是否全覆盖（多条件组合、业务流程、业务规则的需求）
   - 补充缺失的测试方法覆盖

4. **第四步：用例规范性检查（新增）**
   - 检查每条用例的前置条件是否仅包含一种场景/临界值
   - 检查测试名称是否符合"模块-测试点-测试条件-测试结果"格式
   - 检查反向场景是否全面覆盖所有边界情况
   - 检查范围型条件是否拆分为独立用例（小于最小值、等于最小值、中间值、等于最大值、大于最大值）

5. **第五步：整改不符合项**
   - 删除超需求用例
   - 补充缺失的正向/反向用例
   - 合并或删除重复用例
   - 补充测试方法覆盖
   - 补充因果图、场景法、判定表全覆盖的遗漏用例
   - 调整临界值+等价类用例比例至约20%
   - 完善步骤与结果对应关系
   - 提升用例可执行性
   - 拆分混合场景的用例为独立用例
   - 规范测试名称格式
   - 补充缺失的边界值用例

**注意**：自检是强制要求，所有生成的测试用例必须通过自检后才能使用。

## 测试覆盖全面性要求（新增）

为了提升测试用例覆盖的全面性和具体性，生成测试用例时还应考虑以下要求：

### 1. 数据驱动测试要求 ⭐⭐⭐

**要求说明**：
- 明确要求使用多种测试数据（正常数据、边界数据、异常数据）
- 对于输入字段，应该明确要求覆盖不同的数据类型和格式
- 对于选择操作，应该明确要求覆盖所有可选项

**示例**：
- 输入字段：正常值、边界值（最小值、最大值）、异常值（空值、特殊字符、超长字符串）
- 下拉选择：所有选项、默认选项、边界选项

### 2. 界面元素验证要求 ⭐⭐⭐

**要求说明**：
- 明确要求验证界面元素的显示、交互、状态变化
- 包括按钮状态（可用/不可用、显示/隐藏）
- 输入框状态（可编辑/只读、必填/非必填）
- 下拉列表状态（展开/收起、选项显示）
- 表格状态（数据加载、分页、排序）

**示例**：
- 步骤：点击"提交"按钮
- 预期结果：按钮变为不可用状态，显示"提交中..."提示

### 3. 数据一致性验证 ⭐⭐⭐

**要求说明**：
- 明确要求验证数据在不同页面/模块间的一致性
- 验证数据保存后的正确性
- 验证数据修改后的同步性

**示例**：
- 在列表页面创建数据后，验证详情页面数据是否正确
- 修改数据后，验证列表页面数据是否同步更新

### 4. 错误提示验证要求 ⭐⭐⭐⭐

**要求说明**：
- 明确要求验证各种错误场景下的提示信息是否准确、友好
- 包括必填项提示、格式错误提示、业务规则违反提示
- 提示信息的位置、样式、内容准确性

**示例**：
- 步骤：在必填字段留空，点击"提交"按钮
- 预期结果：字段下方显示红色错误提示"该字段为必填项"，提示信息清晰明确

### 5. 权限验证详细要求 ⭐⭐⭐⭐

**要求说明**：
- 明确要求验证不同角色、不同权限下的功能访问和操作限制
- 包括查看权限、编辑权限、删除权限、审批权限等
- 权限不足时的提示信息

**示例**：
- 步骤：普通用户尝试访问管理员功能
- 预期结果：系统提示"您没有权限访问此功能"，功能入口不可见或不可点击

### 6. 业务流程完整性验证 ⭐⭐⭐⭐

**要求说明**：
- 明确要求验证完整业务流程的端到端测试
- 包括数据流转、状态变更、通知提醒等
- 验证流程中每个环节的正确性

**示例**：
- 完整流程：申请 → 审批 → 执行 → 完成
- 验证每个环节的状态变更、数据流转、通知提醒

### 7. 数据边界验证详细规则 ⭐⭐⭐

**要求说明**：
- 明确要求验证数据边界情况（最大值、最小值、空值、特殊字符等）
- 包括输入长度限制、数值范围限制、日期范围限制等
- 边界值的处理方式

**示例**：
- 输入字段限制1-100：测试0、1、2、99、100、101
- 日期范围：测试开始日期前一天、开始日期、结束日期、结束日期后一天

### 8. 性能相关测试点 ⭐⭐（可选）

**要求说明**：
- 对于大数据量场景，应该明确要求验证响应时间、加载速度
- 对于分页、查询等功能，应该明确要求验证性能
- 可以标注为可选测试点，根据实际情况决定是否执行

**示例**：
- 步骤：查询10000条数据
- 预期结果：查询结果在3秒内加载完成，分页功能正常

## CSV格式字段定义（强制）

### 标准字段结构

测试用例CSV文件必须遵循以下8列标准格式：

| 序号 | 字段名称 | 说明 | 必填 |
|------|----------|------|------|
| 1 | 用例ID | 测试用例唯一标识，格式为TC-xxx | 是 |
| 2 | 关联需求 | 关联的需求编号或功能点ID（如：R001,R002） | 是 |
| 3 | 测试场景 | 测试用例名称，格式：模块-测试点-测试条件-测试结果 | 是 |
| 4 | 前置条件 | 执行测试前的准备条件 | 是 |
| 5 | 步骤描述 | 具体操作描述，**多步骤用\n分隔合并在一个单元格中**，带序号（1.xxx\n2.xxx） | 是 |
| 6 | 输入 | 步骤输入数据，**多输入用\n分隔** | 否 |
| 7 | 期望输出 | 预期结果，**多结果用\n分隔合并在一个单元格中**，带序号（1.xxx\n2.xxx） | 是 |
| 8 | 优先级 | 用例优先级：P0、P1、P2 | 是 |

### 字段详细说明

1. **用例ID**：必须唯一，格式为TC-xxx（xxx为数字，从001开始），放在第一列
2. **关联需求**：关联的需求编号或功能点ID（如：R001,R002），放在用例ID之后第二列
3. **测试场景**：必须包含完整测试信息，格式为"模块-测试点-测试条件-测试结果"
4. **前置条件**：描述执行测试前的环境、数据、权限等条件
5. **步骤描述**：具体操作步骤，**多行合并在一个单元格中**，使用`\n`换行，**必须带序号**
6. **输入**：步骤中使用的输入数据，**多输入用\n分隔**
7. **期望输出**：预期结果，**多行合并在一个单元格中**，使用`\n`换行，**必须带序号**
8. **优先级**：P0（最高优先级，核心功能）、P1（重要功能）、P2（一般功能）

### 禁止出现的字段

以下字段**严禁**出现在CSV中：
- 用例关键字
- 关联业务
- 步骤名称
- 备注
- 5cee1b5ae0ee7741889a51f8（或任何类似无意义字段）

### 用户名输入规则

根据PRD要求，用户名必须满足以下规则：
- **长度**：4-50字符
- **字符集支持**：中文、英文、数字、下划线（_）
- **测试用例需覆盖**：
  - 中文用户名测试（如：张三、用户_001）
  - 英文用户名测试（如：testuser）
  - 数字用户名测试（如：user123）
  - 下划线用户名测试（如：user_name）
  - 混合字符测试（如：张三_testuser_001）

### 多行内容处理规则

**步骤描述和期望输出必须是多行合并在一个单元格中，带序号：**

✅ **正确格式**：
```
步骤描述：1.打开登录页面\n2.输入正确的用户名和密码\n3.点击登录按钮
期望输出：1.登录页面正常显示\n2.用户名和密码输入成功\n3.登录成功，跳转到系统首页
```

❌ **错误格式**（缺少序号）：
```
步骤描述：打开登录页面\n输入正确的用户名和密码\n点击登录按钮
期望输出：登录页面正常显示\n登录成功\n跳转到系统首页
```

**示例**：
```
步骤描述：1.打开登录页面\n2.输入正确的用户名和密码\n3.点击登录按钮
期望输出：1.登录页面正常显示，包含用户名和密码输入框\n2.用户名和密码输入成功\n3.登录成功，跳转到系统首页
```

## 注意事项

1. **时间单位为天**：所有时间评估结果以"天"为单位，保留1位小数
2. **综合系数限制**：综合系数限制在0.7~2.0范围内，避免极端值导致评估结果偏离实际
3. **需求复杂度**：需求复杂度会影响多个阶段的基础时间，需要准确评估
4. **熟悉程度变化**：对需求熟悉程度会根据不同需求而变化，需要针对每个需求单独评估
5. **自动化率影响**：自动化率越高，用例执行时间越短，需要根据实际自动化情况设置
6. **执行策略影响**：并行执行可减少总执行时间，串行执行时间更长
7. **BUG严重程度**：严重BUG占比影响修复时间和回归验证时间，需要根据历史数据预估
8. **需求变更风险**：创新型需求变更概率高，需要在总时间基础上增加20%的缓冲
9. **环境因素**：测试环境稳定性、数据准备难度等实际因素可能影响实际执行时间
10. **四舍五入**：最终时间结果四舍五入保留1位小数
